# agent_harness

Claude Code가 이 저장소에서 작업할 때 따르는 협업 규칙과 문서화 워크플로우를 정의하는 저장소입니다.
애플리케이션 코드가 아니라, "어떻게 작업을 위임하고, 검증하고, 기록하는가"를 규정하는 프로세스 정의 모음입니다.

---

## 프로젝트 개요

- 목적: Claude Code와의 협업에서 임의로 코드가 작성되지 않도록, 모든 구현 작업에 사용자 승인 절차를 강제합니다.
- 구성: 최상위 `CLAUDE.md`가 공통 원칙을 정의하고, `.claude/commands/`의 각 문서가 세부 워크플로우를 정의합니다.
- 산출물: 작업 중 생성되는 문서는 모두 `docs/` 하위에 단계별로 쌓이며, 한 작업 사이클이 끝나면 `docs/archive/`로 이동합니다.

---

## 핵심 구성 파일

- `CLAUDE.md` — 협업 원칙, 문서 작성 규칙, 코드 주석 규칙, 문서 아카이빙 규칙, 세션 복구 규칙, 핵심 개발 원칙(SOLID, 외부 라이브러리 래핑, 단위 테스트)을 정의합니다.
- `.claude/commands/task_delegation.md` — 작업 위임의 5단계 절차를 정의합니다.
- `.claude/commands/code_review.md` — 구현 완료 후 2단계 코드 리뷰(품질 검사, 다이어그램 생성) 절차를 정의합니다.
- `.claude/commands/refactor.md` — 코드는 수정하지 않고 리팩토링 후보만 찾아 이슈로 보고하는 절차를 정의합니다.
- `.claude/commands/debug.md` — 커밋 리포트에 기록된 이슈를 재현하고 근본 원인을 분석해 수정하는 절차를 정의합니다.
- `.claude/commands/dependency_eval.md` — 새 외부 라이브러리를 도입하기 전 평가하고 래퍼 클래스를 설계하는 절차를 정의합니다.
- `.claude/settings.local.json` — 허용된 도구/권한 목록을 정의합니다.

---

## 주요 업무 순서 (Task Delegation Workflow)

한 작업은 아래 5단계를 순서대로 거칩니다.

1. **작업 문서 작성** — 목적, 요구사항 등을 확인해 `docs/task/summary_YYYYMMDD_HHMM.md`로 저장합니다.
2. **브레인스토밍 (양방향)** — 목표와 리스크를 정리해 `docs/brainstorming/`에 초안을 작성하고, 사용자 피드백을 받아 승인될 때까지 수정합니다.
3. **전략 문서 작성 (양방향)** — 브레인스토밍 결과를 구현 체크리스트로 구체화해 `docs/strategy/`에 작성하고, 승인 전까지 구현을 시작하지 않습니다.
4. **구현** — 승인된 체크리스트에 따라 구현하며, 항목 완료 시마다 체크리스트를 갱신합니다. 완료 후 코드 리뷰와 리팩토링 검사를 진행할지 사용자에게 확인합니다.
5. **결과 및 이슈 보고** — 구현 결과와 발견된 이슈를 심각도별로 `docs/commit/`에 기록하고, 필요 시 디버그 절차로 연결합니다.

작업 사이클이 모두 끝나면(체크리스트 전체 완료, 이슈 전체 RESOLVED/DEFERRED) 관련 문서를 `docs/archive/summary_YYYYMMDD/`로 이동해 보관합니다.

---

## 문서 구조

```
docs/
  task/            처리 대기 중인 작업 문서
  brainstorming/   승인된 작업의 브레인스토밍 문서
  strategy/        승인된 구현 전략 및 체크리스트
  commit/          구현 결과 및 이슈 리포트
  review/          코드 리뷰 결과 및 다이어그램
  dependency/       외부 라이브러리 평가 결과
  archive/         완료된 작업 사이클 보관함
```

세션을 재개할 때는 `docs/task/`, `docs/brainstorming/`, `docs/strategy/`, `docs/commit/`만 참고하며, `docs/archive/`는 사용자가 명시적으로 요청할 때만 확인합니다.

---

## 개발 원칙

- SOLID 원칙, 특히 단일 책임 원칙(SRP)을 엄격히 적용합니다.
- 외부 라이브러리는 반드시 내부 래퍼 클래스를 통해서만 사용합니다.
- 모든 기능에 단위 테스트를 작성하고, 변경 시마다 빌드와 테스트를 실행합니다.

## Git 워크플로우

브랜치 전략, 커밋/머지 규칙, 릴리즈 태깅 방식은 `.claude/commands/git_workflow.md`를 참고하세요.
